# AGENTS.md — Void Energy UI

Repo-wide instructions for any AGENTS.md-aware coding agent (Aider, goose, opencode, Zed, Warp, JetBrains Junie, Devin, Factory, Jules, RooCode, Kilo Code, Augment, Windsurf, GitHub Copilot coding agent, OpenAI Codex, etc.). This file is loaded at the repo root on session start and is the primary always-on instruction surface for the 24-tool [agents.md](https://agents.md) ecosystem stewarded by the Linux Foundation Agentic AI Foundation.

## Mission

Void Energy UI is a **physics-based, theme-reactive design system** for narrative software. It separates **composition** (what goes where) from **material** (how it looks, moves, and feels). Every surface is the intersection of three environmental variables — an **atmosphere** (palette + font), a **physics** preset (texture, motion, borders), and a **mode** (light or dark polarity) — composed from a token dictionary, never raw values. An optional **brand** overlay can sit between physics and atmosphere to carry identity (radii / motion / type-treatment / per-role weights) without touching color or composition. A runtime engine enforces constraints at the DOM level.

> Do not paint pixels. Declare materials.

The job in this repo is to extend or compose the system without breaking its contract. Prefer small, reversible edits that follow existing patterns. Read local code first; do not invent patterns the repo does not already use.

## Non-Negotiables — The Five Laws

### 1. Hybrid Protocol

Tailwind owns composition (page-level layout, responsive breakpoints, consumer-side geometry). SCSS owns material (shadows, blur, glows, border physics, primitive-internal geometry tied to a token).

- ✅ `class="flex flex-col gap-md p-lg"` — composition in Tailwind
- ✅ `.card { @include surface-raised; }` — material in SCSS
- ❌ `class="shadow-lg bg-blue-500"` — physics in Tailwind
- ❌ `.section { display: grid; gap: 24px; }` — composition in SCSS

Avoid inline styles unless runtime positioning or browser APIs require them.

### 2. Token Law

No raw px, hex, rgb, hsl, or rem values in app code. Every pixel, color, and duration is a semantic token from `src/config/design-tokens.ts`.

- ✅ `padding: var(--space-lg);` `class="gap-md p-lg text-body"`
- ❌ `padding: 32px;` `class="gap-[20px] max-w-[400px]"` `color: #ffffff;`
- **Bracket-syntax Tailwind utilities are banned** — including responsive variants like `tablet:max-w-[400px]`. Compose from the semantic token scale; if no token covers the case, use SCSS with a token, not an arbitrary bracket utility.
- Reviewed exceptions carry `// void-ignore` with a written justification (shimmer highlights, readability floors, browser-mandated constants, scan-line / dotted-pattern stripe widths). They are audited; do not add new ones casually.

### 3. Runes Doctrine — Svelte 5 only

Use runes: `$props`, `$state`, `$derived`, `$effect`, `$bindable`. Do not introduce `export let`, `$:`, `onMount`, `onDestroy`, or `createEventDispatcher`. Prefer callback props (`onchange`, `onclose`, `onsubmit`). Prefer snippets and `{@render ...}` over legacy slots when composition needs structure. Data loading on mount uses `$effect`, not `onMount` — write the rune.

### 4. State Protocol

State lives in the DOM. Expose UI state through `data-*` and ARIA attributes, not modifier classes.

- ✅ `data-state="open"` `aria-pressed="true"` `aria-expanded="false"`
- ❌ `class="is-active"` `class="open"` `class="selected"`

This guarantees enter/exit transitions fire through the Physics Engine and screen readers pick up the same signal the UI uses.

### 5. Spacing Gravity

Default to generous spacing. When uncertain, go ONE size up — never down.

- Floating surface (`.surface-raised`) → `p-lg gap-lg`.
- Sunk well (`.surface-sunk`) → `p-md gap-md`.
- Between page sections → `gap-2xl`. Between content blocks → `gap-xl`. Card grids → `gap-lg`. Form field groups → `gap-md`.
- `gap-xs` and `gap-sm` are reserved for items that are semantically one unit (label→input, icon+text, chip clusters). Never use them between distinct groups.

### Native-First (corollary)

Components are thin wrappers around native HTML elements. Always native: `<button>`, `<input>`, `<select>`, `<textarea>`, `<dialog>`, `<details>`, `<fieldset>`, `<label>`, `<a>`, `<form>`. Custom only when no native element covers the interaction (combobox, multi-thumb slider, virtualized table, command palette, floating panels). Reference: `Selector.svelte` wraps a native `<select>` with label association and SCSS physics — zero custom dropdown JS.

## Component and Styling Conventions

- **SCSS** component files import the toolkit with `@use '../abstracts' as *;`. Default to `var(--radius-base)` for rounded corners and `var(--radius-full)` for pills. Retro physics force radius to zero — do not fight that with hardcoded values.
- **Component APIs** are typed and explicit. Most reusable components include `class?: string` and callback props. Use snippets for structured children when the existing analog does.
- **Buttons.** Variants are class names, not props. Valid variant classes: `btn-cta`, `btn-premium`, `btn-system`, `btn-success`, `btn-error`, `btn-ghost`, `btn-loud`, `btn-icon`, `btn-void`. There is no `variant=` prop on `ActionBtn`, `IconBtn`, `ProfileBtn`, or `ThemesBtn`. Modal-dismiss / cancel uses `class="btn-ghost btn-error"` (red ghost). `ActionBtn` takes `text` as a prop, not children. `IconBtn` is icon-only — pass `icon` plus accessibility-relevant `...rest` (`aria-label`, `title`).
- **Icons.** Static icons come from `@lucide/svelte` with `class="icon"`. Custom interactive icons live in `src/components/icons/` with the `icon-[name] icon` class pattern. Icons inherit color via `currentColor`; set color at the usage site. Use `data-size` for icon sizing.
- **Overlays.** `Modal`, `Dropdown`, and `Sidebar` take Svelte 5 snippets via slot props (`trigger`, `panel`, `children`). Do not call them with positional children where the registry calls for a snippet.
- **`Toggle`** exposes a native `<input type="checkbox" role="switch">`. Use `bind:checked` and `...rest` for native attributes (`name`, `disabled`, `aria-*`).
- **Markdown rendering.** Render markdown-bearing strings through `<Markdown source={...} />`. Plain text goes into a `<p>`. Never hand-roll `marked()` + `{@html}` — the primitive runs `marked` + `sanitize-html` and applies `.prose` styling. The `trusted` flag bypasses the sanitizer and must only be used for source-committed strings (changelog, help copy); when `trusted` appears in a diff, treat it as a sanitizer-bypass review surface.
- **`use:aura`.** Apply only to image-backed or atmosphere-primary surfaces (story scenes, hero panels, album-style cards). Do not attach Aura to dashboard tiles, form fields, or navigation chrome. Multiple Auras in a single visible region produce rainbow-disco output; prefer one focal Aura per region.

### When to build new

Only when **all three** are true: (1) no native element covers the interaction, (2) no shipped primitive covers the behavior, (3) the need will recur in three or more places. Otherwise, compose with what already ships. New consumer-side components belong in `@components/app/`, not `@components/ui/` (`@components/ui/` is reserved for the shipped primitive library and is read-only).

## Runtime Contracts

Import and use the existing singletons. Do not re-instantiate them.

- `voidEngine` from `@adapters/void-engine.svelte` — theme / physics / atmosphere runtime
- `modal` from `@lib/modal-manager.svelte` — open / close modal fragments
- `toast` from `@stores/toast.svelte` — info / success / error / warning + loading + promise
- `layerStack` from `@lib/layer-stack.svelte` — dismissible-layer LIFO (Esc, Back gesture)
- `shortcutRegistry` from `@lib/shortcut-registry.svelte` — keyboard shortcuts
- `user` from `@stores/user.svelte` — auth / user state

Transitions live in `@lib/transitions.svelte` (`emerge`, `dissolve`, `materialize`, `dematerialize`, `implode`, `live`). Actions live under `@actions/*` (`morph`, `tooltip`, `navlink`, `kinetic`, `narrative`, `draggable` / `dropTarget`, `aura`, `laserAim`, `fontShift`).

The `<html>` element carries the runtime state:

- `data-atmosphere` — active theme ID
- `data-physics` — `glass` | `flat` | `retro`
- `data-mode` — `light` | `dark`
- `data-brand` — active brand profile id (only set when atmosphere references one — e.g. `nike`, `stripe`, `lamborghini`)
- `data-auth` — present when any user is authenticated (Guest role included)

Engine-enforced physics invariants (only four of six combinations valid):

- `glass` + `light` → runtime downgrades physics to `flat` (glass needs darkness to glow)
- `retro` + `light` → runtime forces mode to `dark` (CRT phosphor requires a black canvas)
- `flat` works in both modes

Drive the triad through `voidEngine.setAtmosphere()` or DOM attributes. Never patch invalid combinations manually.

### Path Aliases

Valid alias roots — do not invent new ones:

`@components/{ui,core,icons,app}/`, `@adapters/`, `@actions/`, `@lib/`, `@stores/`, `@config/`, `@service/`, `@styles/`. There is no `@components/dashboard/`, `@components/forms/`, etc. Compose pages from the four valid `@components/*` subroots.

### Shared Types

Reusable types live in `src/types/` as ambient global declarations — do not import them. Keep file-local types unexported. If no existing type file fits, create a focused domain file (e.g. `src/types/drag.d.ts`).

## Verification

Run the narrowest command set that actually validates the work:

- `npm run check` — TypeScript, Svelte, registry validation, design-md regenerability, and skill regenerability (covers all generated files in §"Generated Files")
- `npm run test` — unit tests (Vitest, jsdom)
- `npm run scan` — advisory raw-pixel scan after editing `.svelte` or `.scss` files (helper, not a complete Token Law gate)
- `npm run build:tokens` — regenerate `_generated-themes.scss` after editing `src/config/design-tokens.ts`
- `npm run build:skill` — regenerate the seven L2 distribution artifacts after editing `SYSTEM-PROMPT.md` or any `scripts/skill-templates/*.template.md`
- `npm run build` — full production build when changes affect integration or build behavior
- `npm run format` — Prettier across touched files

When UI behavior or styling changes, verify across all three physics presets (`glass`, `flat`, `retro`) and both color modes where the active physics permits (`glass` and `retro` are dark-only).
