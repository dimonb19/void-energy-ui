# Phase 2 — Mobile Deployment (Capacitor)

> Ship Void Energy apps to App Store and Google Play via Capacitor + cloud builds.
> Planned: 2026-04-12. Executes after Phase 1 (AI automation), before Phase 3 (monorepo restructure).

---

## Why This Phase Exists

Void Energy apps are web apps. App Store and Play Store require native binaries. Capacitor bridges that gap: it wraps the compiled web assets in a native WebView shell and exposes device APIs (push, haptics, biometrics, status bar) via a JS bridge. Cloud build services (Capawesome, Appflow, Codemagic) compile and sign the native binaries — **no Mac, no Xcode, no Android Studio required** on the builder's machine.

**Why here, before the monorepo phase:** mobile surfaces load-bearing constraints that the monorepo's package boundaries need to bake in — a `capacitor.ts` helper in `void-energy`, a safe-area utility contract in `@void-energy/tailwind`, and a decision on the default mobile physics preset after glass perf testing. Proving these in the current flat repo first means the Phase 3 monorepo split only has to redistribute files, not invent mobile concepts and restructure simultaneously. It also means the first shippable Void Energy mobile app (the DGRS app or an early CoNexus build) lands months sooner.

This phase delivers two things:

1. **Mobile capability wired into the current `void-energy-ui` repo** — Capacitor integrated, safe areas audited, status bar sync, haptics, glass perf decision made, cloud build pipeline proven end-to-end with a first shipped app.
2. **A documented extraction path** — the Phase 3 monorepo inherits all of this and promotes the working setup into `apps/starter-template/` (or a dedicated `apps/mobile-template/`). The template creation itself is Phase 3 work; Phase 2 just makes sure the content exists and works.

Builders work only in `src/` with Claude Code; native configuration is done once by the repo maintainer (Dimon).

## The App-Split Model

The current `void-energy-ui` runs as SSR (`output: 'server'`) on Vercel. Mobile apps cannot run Node.js in a WebView. The deployment model splits the app:

```
┌─────────────────────────────┐     HTTPS     ┌──────────────────────┐
│  Mobile App (Capacitor)     │ ◄───────────► │  Vercel / Go Backend │
│  ─────────────────────────  │               │  ────────────────────│
│  Astro static build (dist/) │               │  API routes (SSR)    │
│  Svelte components          │               │  AI generation       │
│  Void Energy L1 + L0        │               │  Auth, data, etc.    │
│  WebView (WKWebView / CWV)  │               │                      │
└─────────────────────────────┘               └──────────────────────┘
```

- **Mobile shell:** `output: 'static'` — all pages prerender to HTML/CSS/JS in `dist/`
- **Backend:** existing Go REST API (or Vercel SSR routes) serves data and AI endpoints over HTTPS
- **CORS:** Go/Vercel middleware must whitelist `capacitor://localhost` (iOS) and `https://localhost` (Android)

Apps that have no server-side logic (static content, local-only features) skip the backend entirely.

## Void Energy Integration Points

These are the VE-specific concerns that generic Capacitor guides don't cover.

### 1. Safe Areas (already partially wired)

The reset layer already exposes safe-area tokens:

```scss
// src/styles/base/_reset.scss (lines 89-92)
--safe-top: env(safe-area-inset-top, 0px);
--safe-bottom: env(safe-area-inset-bottom, 0px);
--safe-left: env(safe-area-inset-left, 0px);
--safe-right: env(safe-area-inset-right, 0px);
```

**What's needed:** audit which layout primitives actually consume these tokens. Critical consumers:

| Element | Token needed | Why |
|---------|-------------|-----|
| Fixed top nav / island | `--safe-top` | Dynamic Island, notch |
| Bottom nav island | `--safe-bottom` | Gesture bar |
| Modal overlay | `--safe-top` + `--safe-bottom` | Full-screen dialogs |
| Toast container | `--safe-bottom` | Toasts must clear gesture bar |
| Sidebar | `--safe-left` or `--safe-right` | Landscape edge insets |

The base layout template must include `viewport-fit=cover`:
```html
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover" />
```

### 2. Status Bar Color Sync

The iOS/Android status bar must match the active atmosphere. Without this, switching from Void (dark) to Meridian (light) leaves a black status bar on a white page.

```ts
// In the app's root layout or atmosphere change handler
import { StatusBar, Style } from '@capacitor/status-bar';
import { Capacitor } from '@capacitor/core';

function syncStatusBar(mode: 'dark' | 'light') {
  if (!Capacitor.isNativePlatform()) return;
  StatusBar.setStyle({
    style: mode === 'dark' ? Style.Dark : Style.Light,
  });
}
```

Wire this to `voidEngine`'s mode changes. The exact integration point depends on how the consuming app observes atmosphere switches (likely a `$effect` watching `voidEngine.currentTheme`).

### 3. Glass Physics — The Performance Risk

**This is the biggest unknown.** Glass physics relies heavily on `backdrop-filter: blur()`. The codebase uses it in 6+ SCSS files (16 total occurrences across mixins, dialogs, sidebar, pull-refresh).

- **iOS WKWebView:** `backdrop-filter` is well-supported and GPU-accelerated. Expect glass to work well.
- **Android WebView (Chromium):** Supported since Chrome 76, but mid-range devices (2-3GB RAM, Snapdragon 600-series) can drop to 30fps on large blurred surfaces, especially with overlapping layers.
- **Ambient Layers + glass:** Canvas animations running simultaneously with backdrop-filter blur is the worst-case scenario for GPU memory.

**Required before shipping:**

1. Test glass physics on a mid-range Android device (e.g., Samsung Galaxy A-series, Pixel 6a)
2. Measure frame rate with Chrome DevTools remote debugging via `chrome://inspect`
3. If glass drops below 45fps on target devices, implement one of:
   - **Auto-fallback:** detect low-end hardware → switch to flat physics on mobile
   - **Reduced glass:** fewer blur layers, smaller blur radius on mobile
   - **User choice:** let users pick their physics preset (current architecture already supports this)

**Decision required:** if glass can't sustain 60fps on mid-range Android, flat becomes the mobile default. This is a product decision, not a technical one.

### 4. Haptics Integration

Route haptic feedback through VE's existing action/button system so it's a single integration point:

```ts
import { Haptics, ImpactStyle } from '@capacitor/haptics';
import { Capacitor } from '@capacitor/core';

export async function hapticTap() {
  if (!Capacitor.isNativePlatform()) return;
  await Haptics.impact({ style: ImpactStyle.Light });
}

export async function hapticSuccess() {
  if (!Capacitor.isNativePlatform()) return;
  await Haptics.notification({ type: 'SUCCESS' });
}
```

Logical integration points: `ActionBtn` press, `Toggle` switch, `Modal` open/close, `Toast` appear. Should be opt-in via a VE user preference (`voidEngine.userConfig.haptics`).

### 5. Keyboard Behavior

Capacitor's `Keyboard.resize: 'ionic'` is critical for VE apps because:
- Default `native` resize breaks `100dvh` layouts
- VE modals use fixed positioning with `--safe-bottom` — keyboard must push content, not overlap
- `EditField`, `PasswordField`, `GenerateField` all need the keyboard to push the viewport

### 6. Pull-to-Refresh

The browser's native pull-to-refresh must be disabled in the WebView — it conflicts with VE's custom `_pull-refresh.scss`. Capacitor handles this via:

```ts
// capacitor.config.ts → ios
ios: {
  scrollEnabled: false, // Disable bounce/pull-refresh at WebView level
}
```

VE's own pull-to-refresh component (if used) handles the gesture internally.

### 7. Ambient Layers Budget

Canvas-based ambient effects (Blood, Snow, Rain, Fog) have a GPU cost. On mobile:
- Limit to 1 active ambient layer at a time
- Auto-disable ambient layers when battery is below 20% (via `@capacitor/device` or Battery API)
- Provide a `reducedMotion` preference that disables ambient layers entirely
- Respect `prefers-reduced-motion` media query (already in VE's accessibility layer)

### 8. App Lifecycle

VE animations and effects should pause when the app is backgrounded:

```ts
import { App } from '@capacitor/app';

App.addListener('appStateChange', ({ isActive }) => {
  if (!isActive) {
    // Pause ambient layers, kinetic text, requestAnimationFrame loops
  } else {
    // Resume
  }
});
```

This prevents battery drain and avoids the "animation jumped 30 seconds forward" artifact when returning to the app.

### 9. Font Loading

Web fonts load over the network on first visit. In a Capacitor app, fonts should be bundled:
- Include font files in `public/fonts/` (they'll be in `dist/` after build)
- Use `@font-face` with relative paths, not CDN URLs
- VE's font registry (`src/config/font-registry.ts`) must resolve to local paths

### 10. Routing

Astro's file-based routing works in Capacitor, but:
- No absolute URLs (`https://yourdomain.com/page`) — use relative paths (`/page`)
- The `navlink` action works as-is (it sets loading state on click, doesn't manipulate URLs)
- Deep links (opening the app from a URL) require Capacitor's App Links / Universal Links plugin — configure per app, not in the template

## Cloud Build Pipeline

The template uses cloud build services so builders never install Xcode or Android Studio.

### Recommended: Capawesome Cloud

```bash
# One-time setup
npm install -g @capawesome/cli
capawesome login
capawesome apps:create --name "App Name" --platform ios,android

# Every deploy
npm run build                    # Astro builds to dist/
npx cap sync                     # Copies dist/ into native projects
capawesome build --platform ios   # Cloud compiles + signs → .ipa
capawesome build --platform android  # Cloud compiles + signs → .aab
```

Signing certificates and provisioning profiles are configured once in the Capawesome web console using your Apple Developer / Google Play credentials.

### Alternatives

- **Appflow (by Ionic):** similar workflow, `ionic deploy` commands, free tier for small teams
- **Codemagic:** more general-purpose CI/CD, supports Capacitor via `codemagic.yaml`

## Open Questions

1. **Glass on Android:** Is flat the mobile default, or do we ship glass and let users downgrade? Needs device testing. **Decision gate before Phase 3.**
2. **Push notification service:** Firebase Cloud Messaging (free, Google-dependent) or a self-hosted alternative?
3. **Which app ships first?** The DGRS Labs mobile app (Go backend, ready) or an early CoNexus mobile build? Affects which backend CORS config lands first.
4. **SSR routes in the current repo:** do the existing `/api/*` routes move to the Go backend now, or stay Vercel-SSR behind `export const prerender = false` via `output: 'hybrid'`?
5. **Offline support:** Should the template include a service worker for offline access, or is that app-specific?
6. **App review guidelines:** Apple's WebView-app rules (4.2 guideline) require "sufficient native functionality" — push notifications + haptics + status bar integration likely satisfy this, but worth confirming with a first submission.
